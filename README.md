# CheckAI prompt lab

Рабочий набор для конкурса CheckAI: промпты, правила итераций и заметки по моделям.

## Быстрый старт для работы

Если ты впервые открыл репозиторий, начинай отсюда:

1. Открой файл `prompts/chinese_v2_prompts.md`.
2. Если трек `Генерация с нуля`, бери блок `A. Universal generation v2`.
3. Если трек `Переписывание фрагмента`, бери блок `B. Universal rewriting v2`.
4. Подставь вместо плейсхолдеров данные с сайта:
   - `{DOMAIN}` - область из списка, например `Аддитивные технологии`;
   - `{TOPIC}` - полная тема или название варианта;
   - `{FRAGMENT}` - исходный фрагмент для трека переписывания;
   - `{SECTION}` - секция исходного фрагмента, если она известна;
   - `{SCORE_FEEDBACK}` - оценки прошлой отправки, если делаешь новый вариант после провала.
5. Вставь готовый промпт в веб-чат модели.
6. Полученный русский текст перенеси в DOCX без смысловых правок.
7. Загрузи DOCX на CheckAI.
8. Запиши результат в таблицу по шаблону `data/results_template.csv`.
9. Обязательно сохрани ссылку на чат с моделью.

Главное правило: не редактировать смысл текста руками. Если результат плохой, генерируем новый вариант через новый запрос или новый чат.

## Цель

Получать сильные полностью AI-сгенерированные DOCX-документы для двух треков:

- `generation`: статья с нуля.
- `rewriting`: переписывание заданного фрагмента.

Важно: отправляемый текст должен быть создан через разрешенный веб-интерфейс модели. Не используем API, локальные LLM и ручное смысловое редактирование текста.

## Где что лежит

- `prompts/chinese_v2_prompts.md` - основные китайские промпты v2. Использовать первыми.
- `prompts/checkai_prompts.md` - старый набор промптов v1. Держим как запасной.
- `planning/topic_strategy.md` - как выбирать стиль под разные группы тем.
- `data/results_template.csv` - шаблон таблицы результатов.
- `docs/operational_checklist.md` - чеклист перед генерацией, после генерации и после отправки.
- `docs/research_notes.md` - заметки по правилам, моделям и детекторам.
- `docs/github_prompt_scan.md` - что было проверено по GitHub-идеям для промптов.

## Какой промпт выбрать

| Ситуация | Файл | Блок |
| --- | --- | --- |
| Первая генерация статьи с нуля | `prompts/chinese_v2_prompts.md` | `A. Universal generation v2` |
| Переписывание фрагмента | `prompts/chinese_v2_prompts.md` | `B. Universal rewriting v2` |

## Базовый цикл

1. Выбрать тему и трек на сайте CheckAI.
2. Открыть новый чат в одной из веб-моделей: DeepSeek, Kimi, Qwen Studio или Z.ai/GLM.
3. Вставить подходящий промпт из `prompts/chinese_v2_prompts.md` или `prompts/checkai_prompts.md`.
4. Получить ответ, который содержит только готовый текст документа или секции.
5. Перенести ответ в DOCX без смысловых правок. Разрешены только технические действия: убрать лишний markdown, жирный шрифт, пустые строки, очевидные опечатки.
6. Отправить DOCX на CheckAI.
7. Записать в таблицу: модель, режим, трек, тема, время, ссылка на чат, `final_score`, `ai_score`, `quality_score`, `topic_relevance_score`, `structure_score` или `section_match_score`.
8. Если результат слабый, не править документ руками. Запустить новый чат или новый запрос и сгенерировать новый независимый вариант.

## Как делать DOCX

1. Скопировать только финальный русский текст из ответа модели.
2. Вставить в Word или другой редактор DOCX.
3. Убрать только технический мусор, если модель добавила лишний markdown, жирность или пустые строки.
4. Не добавлять свои абзацы, источники, пояснения или исправления по смыслу.
5. Проверить, что объем больше 4000 символов.
6. Для генерации с нуля проверить наличие точных секций:
   - `Введение`
   - `Теоретическая часть / Методы`
   - `Практическая часть / Эксперименты, результаты`
   - `Заключение`
7. Сохранить файл с понятным именем, например `generation_additive_microcracks_deepseek_v1.docx`.

## Что сохранять для валидации

Для каждой работы сохранить:

- ссылку на чат;
- модель и режим;
- дату и время генерации;
- какой промпт использовался;
- имя DOCX;
- скрин или экспорт чата, если веб-модель не дает нормальную ссылку.

Это важно: топ-5 будут просить отчет с историей чатов. Если ссылок нет, результат могут не засчитать.

## Как читать провал

- Низкая релевантность: в следующем промпте добавить точную формулировку темы в первые 2 абзаца и больше предметных терминов.
- Плохая структура: использовать точные заголовки из правил.
- Низкое качество: добавить метод, материал, наблюдения, ограничения и конкретный результат.
- Высокий `ai_score`: просить менее симметричную композицию, разную длину абзацев, меньше общих переходов, больше предметной конкретики.

## Приоритет моделей

1. DeepSeek Expert / Thinking: основной кандидат для первого прогона.
2. Kimi K2.6 Thinking: хорошо держит длинный контекст и сложные ограничения.
3. Qwen Studio / Qwen3.6-Plus: сильный запасной вариант, особенно для русского после китайского управляющего промпта.
4. GLM / Z.ai: тестировать как четвертую ветку, особенно на темах с инженерной или методологической структурой.

## Какие промпты брать первыми

1. Для генерации с нуля начать с `prompts/chinese_v2_prompts.md`, блок `A. Universal generation v2`.
2. Если DeepSeek дает слишком ровную лекцию, использовать блок `C. DeepSeek strict v2`.
3. Для Kimi брать блок `D. Kimi long-context v2`.
4. Для Qwen брать блок `E. Qwen clear-instruction v2`.
5. Для переписывания фрагмента брать блок `B. Universal rewriting v2`.
6. После слабой отправки брать блок `F. Score repair v2` и делать новый независимый вариант.

## Минимальная стратегия лидерборда

Для входа в лидерборд нужно 5 тем, но формула считает 10 лучших работ, а недостающие идут как нули. Поэтому рабочая цель: минимум 10 сильных документов на трек, лучше больше, если дневной лимит позволяет.

## Минимальный план на день

1. Выбрать 5 тем в одном треке.
2. На каждую тему сделать первый прогон через DeepSeek Expert и блок `A`.
3. Отправить DOCX, записать оценки.
4. Для слабых тем сделать второй прогон:
   - DeepSeek block `C`, если текст был слишком гладкий;
   - Kimi block `D`, если нужна более строгая структура;
   - Qwen block `E`, если хочется другой стиль.
5. После первых 5 тем расширяться до 10 тем.
6. В лидерборд выбирать лучшие документы по разным темам.

## Как заполнять таблицу

Шаблон лежит в `data/results_template.csv`.

Обязательные поля:

- `date` - дата генерации или отправки;
- `track` - `generation` или `rewriting`;
- `domain` - область;
- `topic` - полная тема;
- `model` - DeepSeek, Kimi, Qwen или GLM;
- `mode` - Expert, Thinking, Auto и т.п.;
- `prompt_version` - например `chinese_v2_A`;
- `doc_name` - имя DOCX;
- `chat_link` - ссылка на чат;
- `final_score` - итоговая оценка;
- `ai_score` - оценка детектора;
- `quality_score` - качество;
- `topic_relevance_score` - релевантность;
- `structure_score` или `section_match_score`;
- `char_count` - символы;
- `notes` - что просело;
- `next_action` - что делать дальше.

## Что делать нельзя

- Нельзя использовать API.
- Нельзя использовать локально развернутые модели.
- Нельзя прогонять текст через humanizer-сервисы.
- Нельзя вручную переписывать, удалять или добавлять смысловые части.
- Нельзя отправлять одинаковые или почти одинаковые документы на разные темы.
- Нельзя терять ссылки на чаты.
